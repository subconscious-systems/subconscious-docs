Streaming delivers the agent's response token-by-token as it's generated—ideal for chat interfaces, live demos, or anywhere a human is watching. Instead of waiting for the full response, you display output progressively for a more responsive experience.

For background jobs or backend integrations, see [Runs](/core-concepts/runs) for async patterns with polling and webhooks.

## Basic Usage

<CodeGroup>

```typescript Node.js
import { Subconscious, tools } from "subconscious";

const client = new Subconscious({
  apiKey: process.env.SUBCONSCIOUS_API_KEY!,
});

const stream = client.stream({
  engine: "tim",
  input: {
    instructions: "Write a short essay about space exploration",
    tools: [tools.platform("web_search")],
  },
});

for await (const event of stream) {
  switch (event.type) {
    case "started":
      console.log("runId:", event.runId);
      break;
    case "delta":
      process.stdout.write(event.content);
      break;
    case "result":
      console.log("\n\nfinal answer:", event.result.answer);
      break;
    case "error":
      console.error(`[${event.code}] ${event.message}`);
      break;
    case "done":
      console.log("\nstream done.");
      break;
  }
}
```

```python Python
from subconscious import Subconscious, tools
from subconscious.types import (
    StartedEvent, DeltaEvent, ResultEvent, ErrorEvent, DoneEvent,
)

client = Subconscious(api_key="your-api-key")

for event in client.stream(
    engine="tim",
    input={
        "instructions": "Write a short essay about space exploration",
        "tools": [tools.platform("web_search")],
    },
):
    if isinstance(event, StartedEvent):
        print("run_id:", event.run_id)
    elif isinstance(event, DeltaEvent):
        print(event.content, end="", flush=True)
    elif isinstance(event, ResultEvent):
        print("\n\nfinal answer:", event.result.answer)
    elif isinstance(event, ErrorEvent):
        print(f"[{event.code}] {event.message}")
    elif isinstance(event, DoneEvent):
        print("\nstream done.")
```

## Event Types

The stream uses **Stream Events v2** — a structured taxonomy with strict
ordering invariants. The first event is always `started` (it carries
the `runId` synchronously so you can register a cancel handler before
any deltas arrive). The last event is always `done`. Exactly one
`result` (success) or `error` (failure) event fires before `done`.

| Event Type        | Description                                                          |
| ----------------- | -------------------------------------------------------------------- |
| `started`         | First event. Carries the `runId`.                                    |
| `delta`           | Incremental text content from the model.                             |
| `reasoning_node`  | One completed node from the live reasoning tree (engines that emit). |
| `tool_call`       | One completed tool invocation.                                       |
| `result`          | Final structured envelope (answer + reasoning). Exactly once on success. |
| `error`           | Stream encountered an error. Carries `code` and `message`. Exactly once on failure. |
| `done`            | Stream finished. Always last.                                        |

## Server-Sent Events (SSE)

The streaming endpoint uses Server-Sent Events (SSE) framing. The
canonical wire format uses named event types with camelCase `runId`:

```
event: started
data: {"runId": "run_abc123"}

event: meta
data: {"runId": "run_abc123"}

data: {"choices": [{"delta": {"content": "The history of space exploration..."}}]}

event: reasoning_node
data: {"node": {...}}

event: tool_call
data: {"call": {...}}

event: result
data: {"result": {"answer": "..."}, "usage": {...}}

data: [DONE]
```

<Note>
  SDKs running against older API builds will also see legacy snake_case
  `run_id` payloads — the parsers accept both keys. New emitters always
  write `runId`.
</Note>

## Error Handling

Stream errors arrive as a single `error` event before `done`. Pattern-match
on `event.code` (a canonical [`ErrorCode`](/guides/error-handling#error-codes))
rather than parsing `event.message`.

<CodeGroup>

```typescript Node.js
try {
  for await (const event of stream) {
    if (event.type === "delta") {
      process.stdout.write(event.content);
    } else if (event.type === "error") {
      console.error(`[${event.code}] ${event.message}`);
      // event.code is one of: invalid_request | authentication_failed |
      // permission_denied | not_found | rate_limited | internal_error |
      // service_unavailable | timeout | canceled
      break;
    }
  }
} catch (error) {
  console.error("Connection error:", error);
}
```

```python Python
from subconscious.types import DeltaEvent, ErrorEvent

try:
    for event in client.stream(...):
        if isinstance(event, DeltaEvent):
            print(event.content, end="", flush=True)
        elif isinstance(event, ErrorEvent):
            print(f"[{event.code}] {event.message}")
            break
except Exception as e:
    print(f"Connection error: {e}")
```
