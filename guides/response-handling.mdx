When a run completes, the agent returns structured results that include both the final answer and the reasoning process. This guide explains how to interpret and use these responses.

## Response Structure

The `run.result` object contains:

```typescript
interface RunResult<T = string> {
  answer: T; // string for free-form runs; the structured value when answerFormat is set
  reasoning: ReasoningNode[] | null; // step-by-step reasoning tree (when available)
}
```

## The Answer Field

The shape of `result.answer` depends on whether you set `answerFormat`:

- **Without `answerFormat`**: a natural language string.
- **With `answerFormat`**: the structured value (a Pydantic model / dict in Python; an object typed by the generic on `runAndWait<T>` / `run<T>` / `stream<T>` in TypeScript).

### Accessing Structured Answers

When `answerFormat` is set, the SDKs decode the structured payload for
you — `run.result.answer` is already the decoded value, not a JSON
string. The SDKs do not validate the decoded value against your schema
— validate yourself with Pydantic or Zod if you want a typed instance.

<CodeGroup>

```python Python
from subconscious import Subconscious
from pydantic import BaseModel

class SentimentAnalysis(BaseModel):
    sentiment: str
    confidence: float

client = Subconscious(api_key="your-api-key")

run = client.run_and_wait(
    engine="tim",
    input={
        "instructions": "Analyze sentiment",
        "answerFormat": SentimentAnalysis,
    },
)

# run.result.answer is already a dict matching SentimentAnalysis
print(run.result.answer["sentiment"])

# Hydrate into a typed Pydantic instance if you want validation:
typed = SentimentAnalysis.model_validate(run.result.answer)
print(typed.sentiment)
```

```typescript Node.js
import { Subconscious } from "subconscious";
import { z } from "zod";

const client = new Subconscious({
  apiKey: process.env.SUBCONSCIOUS_API_KEY!,
});

const SentimentAnalysis = z.object({
  sentiment: z.string(),
  confidence: z.number(),
});
type SentimentAnalysis = z.infer<typeof SentimentAnalysis>;

const run = await client.runAndWait<SentimentAnalysis>({
  engine: "tim",
  input: {
    instructions: "Analyze sentiment",
    answerFormat: SentimentAnalysis,
  },
});

// `result.answer` is narrowed to SentimentAnalysis by the generic
console.log(run.result?.answer.sentiment);

// Optionally re-validate at runtime:
// const validated = SentimentAnalysis.parse(run.result?.answer);
```

</CodeGroup>

## Understanding the Reasoning Field

The `reasoning` field provides visibility into how the agent arrived at its answer. Each task in the array represents a step in the reasoning process.

### ReasoningNode Structure

```typescript
interface ReasoningNode {
  title: string;       // Step title
  thought: string;     // Agent's internal reasoning
  tooluse: ToolUse | null;  // Tool call details (at most one per node)
  subtasks: ReasoningNode[]; // Nested reasoning steps
  conclusion: string;  // Step conclusion
}

interface ToolUse {
  tool_name: string;
  parameters?: unknown;
  tool_result?: unknown;
}
```

### Example Reasoning Trace

```json
{
  "reasoning": [
    {
      "title": "Analyzing the request",
      "thought": "I need to search for information about Tesla's stock performance",
      "tooluse": {
        "tool_name": "web_search",
        "parameters": {
          "query": "Tesla stock performance this week"
        },
        "tool_result": {
          "results": [
            {
              "title": "Tesla Stock Analysis",
              "snippet": "Tesla's stock has shown..."
            }
          ]
        }
      },
      "conclusion": "Found relevant information about Tesla's stock"
    }
  ],
  "answer": "Based on my search, Tesla's stock performance this week shows..."
}
```

## Extracting Tool Call Information

Loop through the reasoning to extract tool usage:

<CodeGroup>

```python Python
def extract_tool_calls(reasoning):
    # reasoning is a list of ReasoningTask pydantic models — use attribute access
    tool_calls = []
    for task in reasoning or []:
        if task.tooluse:
            tool_calls.append({
                "tool": task.tooluse.tool_name,
                "params": task.tooluse.parameters,
                "result": task.tooluse.tool_result,
            })
        # Check subtasks recursively
        if task.subtasks:
            tool_calls.extend(extract_tool_calls(task.subtasks))
    return tool_calls

# Usage
calls = extract_tool_calls(run.result.reasoning)
for call in calls:
    print(f"Tool: {call['tool']}")
```

```typescript Node.js
import type { ReasoningNode } from "subconscious";

function extractToolCalls(reasoning: ReasoningNode[]): Array<{
  tool: string;
  params: unknown;
  result: unknown;
}> {
  const toolCalls = [];
  for (const node of reasoning ?? []) {
    if (node.tooluse) {
      toolCalls.push({
        tool: node.tooluse.tool_name,
        params: node.tooluse.parameters,
        result: node.tooluse.tool_result,
      });
    }
    toolCalls.push(...extractToolCalls(node.subtasks));
  }
  return toolCalls;
}

const calls = extractToolCalls(run.result?.reasoning ?? []);
calls.forEach((call) => console.log(`Tool: ${call.tool}`));
```

</CodeGroup>

## Usage Information

The `usage` field provides metrics about the run:

```typescript
interface RunUsage {
  inputTokens: number; // Tokens in the prompt
  outputTokens: number; // Tokens generated
  durationMs?: number; // Total execution time (when reported by the server)
}
```

In Python, fields are exposed as snake_case attributes (`input_tokens`, `output_tokens`, `duration_ms`) on the `Usage` pydantic model.

### Tracking Costs

Use the usage information to track costs:

```python
usage = run.usage
input_cost = usage.input_tokens / 1_000_000 * 2.00  # tim pricing
output_cost = usage.output_tokens / 1_000_000 * 8.00
total_cost = input_cost + output_cost
print(f"Run cost: ${total_cost:.4f}")
```
