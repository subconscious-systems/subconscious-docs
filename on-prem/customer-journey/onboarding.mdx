Onboarding turns the evaluation plan into a working customer-hosted deployment. A Subconscious FDE works with your technical champion and admin team to plan the deployment, stand up the runtime plane, configure the gateway, connect coding agents, and hand off day-two operations.

This page gives the medium-detail journey. For implementation details, use [Methods](/on-prem/deployment-models/methods), [Configurations](/on-prem/deployment-models/configurations), and [Gateway setup](/on-prem/deployment-models/gateway-setup).

## Summary

Typical onboarding expectations:

- Kickoff to first pilot use can often happen in about two weeks when customer technical resources are available.
- Customer configuration time is usually less than a day of focused technical work when the environment is ready.
- Your assigned FDE can join working sessions or come onsite when that helps accelerate setup.
- Admin training and broader team training are designed to fit into short sessions.
- Broader team adoption usually follows a pilot once access, limits, routes, monitoring, and support workflows are validated.

## 1. Kickoff

Kickoff establishes communication channels, confirms the implementation timeline, and identifies owners.

What happens:

- A dedicated Slack channel or customer-approved support channel is created.
- A Subconscious FDE becomes the primary onboarding point of contact.
- The customer identifies the technical champion, admin team, security reviewers, and pilot sponsor.
- The team confirms success criteria, timeline, and responsibilities from evaluation.

## 2. Deployment planning

Deployment planning confirms how the runtime plane will run in your environment.

Topics usually include:

- Cloud provider, region, Kubernetes environment, and namespace.
- GPU sourcing and placement.
- Endpoint strategy, DNS, TLS, ingress, and network policy.
- Deployment method: assisted self-managed or traditional self-hosted.
- Distr customer portal, artifacts, deployment agent, registry access, and secrets.
- Observability, logging, alerting, and support access expectations.

For the detailed decision points, see [Methods](/on-prem/deployment-models/methods), [Distribution platform](/on-prem/deployment-models/distribution-platform), and [Configurations](/on-prem/deployment-models/configurations).

## 3. Runtime plane deployment

The runtime plane is the production inference system your engineers use. It typically includes the gateway, routing layer, model-serving configuration, admin dashboard, observability, and baseline services required for coding-agent traffic.

Customer responsibilities usually include:

- Provisioning or approving required infrastructure and GPU resources.
- Approving deployment commands, agent installation, or internal GitOps changes.
- Confirming operational ownership after onboarding.

Subconscious responsibilities usually include:

- Guiding deployment and configuration.
- Validating runtime health.
- Preparing the gateway for admin setup and pilot traffic.

## 4. Gateway setup and admin training

Once the runtime plane is reachable, the admin team configures the system so pilot users can use it safely.

This includes:

- Admin users and pilot users.
- Gateway API keys.
- Model routes to SGL workers or external model endpoints.
- Model access grants.
- Organization-level and API-key-level limits.
- Test requests through the gateway.
- Usage, monitoring, logging, and operational workflows.

The detailed setup guide lives in [Gateway setup](/on-prem/deployment-models/gateway-setup).

## 5. Coding agent integration and team training

After the gateway is configured, pilot users connect coding agents to the customer endpoint.

Training usually covers:

- What the system does and how it fits into the customer environment.
- Gateway base URL, API keys, and model slugs.
- OpenAI-compatible configuration for coding agents and internal tools.
- Expectations for limits, acceptable use, support, and feedback.

The first rollout usually starts with a small pilot group, then expands after usage, reliability, and route behavior are validated.

## 6. Handoff

At the end of onboarding, Subconscious and the customer confirm that the deployment is ready for day-two operations.

Handoff usually confirms:

- Customer admins can sign in and operate the gateway.
- API keys, routes, access, limits, and test requests are working.
- Pilot users can connect coding agents.
- Monitoring and logging ownership is clear.
- Support and escalation paths are understood.
- Upgrade, rollback, and patch expectations are documented.

After handoff, the relationship continues through [Customer success](/on-prem/customer-journey/customer-success).
