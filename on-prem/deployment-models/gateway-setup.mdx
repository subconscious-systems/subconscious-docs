After the runtime plane is deployed, the next step is gateway setup. This is where your team and Subconscious configure the admin surfaces, users, API keys, SGL worker routes, model access, limits, monitoring, and day-two workflows that make the deployment usable by your engineering organization.

Deployment answers: "How does the software get installed into our environment?"

Gateway setup answers: "How do we configure the installed system so engineers can use it safely?"

## Where this fits

In a typical onboarding:

1. Your Subconscious FDE provisions your customer portal in Distr.
2. Your team chooses [traditional self-hosted or assisted self-managed](/on-prem/deployment-models/methods).
3. The runtime plane is deployed into your cloud or customer-controlled environment.
4. Your FDE helps administrators complete initial gateway setup.
5. Pilot users connect their coding agents to the customer endpoint.
6. Your platform or engineering team takes over day-to-day administration.

## Admin surfaces

There are two primary admin surfaces during onboarding.

### Distr customer portal

The Distr customer portal is used for deployment and distribution workflows.

Use it to:

- View deployment instructions.
- Inspect available artifacts.
- Generate registry access tokens.
- Manage deployment secrets when enabled.
- Review release notes and version resources.
- Review vulnerability reports when attached to releases.
- View deployment status in assisted self-managed mode.

### Subconscious gateway dashboard

The Subconscious gateway dashboard is used for runtime administration.

Use it to:

- Manage organization members.
- Invite administrators or pilot users.
- Create and revoke gateway API keys.
- Configure model routes to SGL workers or external model endpoints.
- Grant model access.
- Configure organization-level or API-key-level limits.
- Review usage.
- Test a request through the gateway.
- Review operational pages exposed by the runtime.

## Prerequisites

Before gateway setup begins, confirm that:

- The customer runtime plane has been deployed.
- The gateway dashboard is reachable from the approved network.
- The initial administrator account has been bootstrapped or invited.
- Customer endpoint DNS and TLS settings are configured.
- SGL workers or external model endpoints are reachable from the gateway.
- Required provider credentials and secrets are available.
- The customer has identified first admin users and pilot users.
- The customer has identified initial model routes and default coding-agent configuration.

## Initial setup checklist

The initial setup usually follows this order:

1. Confirm dashboard access.
2. Invite admin members.
3. Configure model provider credentials and secrets.
4. Create model routes to SGL workers.
5. Grant model access to the customer organization.
6. Create pilot API keys.
7. Configure limits.
8. Send a test request through the dashboard.
9. Configure coding agents for pilot users.
10. Review monitoring, usage, logs, and support workflows.

## Step 1: confirm dashboard access

Verify that the runtime plane is healthy and that customer administrators can reach the gateway dashboard.

Click flow:

1. Open the customer gateway dashboard URL.
2. Sign in with the bootstrap administrator account or customer-approved invite.
3. Confirm the dashboard loads without network, TLS, or authentication errors.
4. Open the overview page and confirm the organization name is correct.
5. Confirm the active organization is the customer production organization.

Validation:

- Dashboard loads over the approved URL.
- TLS certificate is valid.
- Admin user can sign in.
- Organization name and role are correct.
- No critical health warnings are visible.

## Step 2: invite administrators and pilot users

Give the right customer users access to the gateway dashboard and separate admin users from pilot users where applicable.

Invite flow:

1. In the gateway dashboard, open **Members**.
2. In **Invite member**, enter the user's email address.
3. Choose the appropriate role.
4. Click **Create invite link**.
5. Send the invite link through the customer's approved channel.
6. Confirm the user appears in **Recent invitations**.
7. After the user accepts, confirm the user appears in **Members**.

Password reset flow:

1. Open **Members**.
2. In **Create password reset link**, select the member.
3. Click **Create reset link**.
4. Share the reset link through the customer's approved channel.

## Step 3: configure secrets and provider credentials

Store only the credentials needed for the gateway to reach SGL workers or external model endpoints. Keep secrets under the customer's control.

Credential types may include:

- SGL worker credentials if required.
- External model provider API keys.
- Internal service credentials.
- Observability export credentials.
- TLS certificate references.
- Database or cache credentials if customer-managed.

Configuration paths:

- In assisted self-managed mode, deployment secrets may be configured through the Distr customer portal when enabled.
- In traditional self-hosted mode, customers may prefer Kubernetes Secrets, an external secrets operator, or their cloud secret manager.
- In the gateway dashboard, route-specific provider API keys can be entered when creating or editing a route, if that route requires a bearer token or provider credential.

## Step 4: configure routes to SGL workers

Routes tell the gateway which model slug should route to which SGL worker or model endpoint.

Example:

- A coding agent requests `subconscious-coder`.
- The gateway maps `subconscious-coder` to an SGL worker endpoint.
- The gateway applies authentication, access policy, limits, metering, and routing behavior.
- The SGL worker serves the model response.

Create flow:

1. In the gateway dashboard, open **Routes**.
2. In **Create route**, enter **Model slug**.
3. Enter **Provider**.
4. Enter **Route URL** for the SGL worker or model endpoint.
5. Choose **Kind**.
6. Enter **Context** length.
7. Enter **Provider API key** if the target route requires one.
8. Click **Create route**.
9. Confirm the route appears in **Routes**.
10. Confirm **Health** and **Status** are correct.

Current route fields:

- **Model slug**: The model name that coding agents and API clients request.
- **Provider**: A provider label such as `local`, `sgl`, or the external provider name.
- **Route URL**: The reachable URL for the SGL worker or model endpoint.
- **Kind**: The target API compatibility mode, such as OpenAI-compatible routing.
- **Context**: The context length advertised or enforced for the route.
- **Provider API key**: Optional bearer token or provider credential for the target endpoint.
- **Health**: Operational status for the route.
- **Active**: Whether the route is eligible to receive traffic.

Edit flow:

1. Open **Routes**.
2. Click the route row.
3. Update the model slug, provider, route URL, kind, context, provider API key, health, or active status.
4. Click **Save route**.

Delete flow:

1. Open **Routes**.
2. Click the route row.
3. Choose the delete action.
4. Confirm the deletion.

## Step 6: create pilot API keys

Gateway API keys are the credentials that coding agents use to call the customer gateway endpoint.

Create flow:

1. In the gateway dashboard, open **API keys**.
2. Click or use **Create key**.
3. Enter a key **Name**.
4. Choose the appropriate scope if prompted.
5. Click **Create key**.
6. Copy the generated key once and store it in the customer's approved secret manager.
7. Confirm the key appears in the active key list.

Key management workflows:

- Filter keys by active or revoked status.
- Rename keys when ownership changes.
- Revoke keys that are no longer needed.
- Create separate keys for pilot users, teams, CI systems, or shared test environments.

Recommended starting pattern:

- Create one admin test key.
- Create one pilot team key or one key per pilot user, depending on customer policy.
- Avoid broad, long-lived shared keys for production rollout.
- Store keys in the customer's password manager or secret manager.

## Step 7: configure limits

Limits protect the deployment from unexpected usage and align capacity with the pilot or production rollout plan.

Click flow:

1. In the gateway dashboard, open **Limits**.
2. In **Set limit**, choose **Scope**.
3. Choose **Organization** or **API key**.
4. Select an **API key** when using key-scoped limits.
5. Choose **Model** or leave it as all models.
6. Choose **Kind**: requests or tokens.
7. Choose **Window**: minute, hour, day, or month.
8. Enter **Limit**.
9. Click **Set limit**.
10. Confirm the limit appears in **Limits**.

Recommended starting pattern:

- Set conservative pilot limits for the first user group.
- Set higher organization-level limits after performance validation.
- Use model-specific limits when route capacity differs by model.
- Review usage before broad rollout.

## Step 8: send a test request

Send a test request to confirm that authentication, model access, route configuration, and worker connectivity are working end to end.

Click flow:

1. In the gateway dashboard, open **Chat** or the test request page.
2. Choose a **Gateway API key**.
3. Choose a **Model**.
4. Enter a small test prompt.
5. Click **Send**.
6. Confirm the gateway returns a response.
7. Confirm usage is recorded.

Validation:

- API key is accepted.
- Model appears in the model selector.
- Route is active.
- Worker returns a response.
- Usage and token counts are recorded.
- No secrets or sensitive prompt data are exposed in logs.

## Step 9: configure coding agents

Point pilot users' coding agents at the customer gateway endpoint.

Configuration inputs:

- Gateway base URL.
- Gateway API key.
- Model slug.
- Timeout settings.
- Any client-specific OpenAI-compatible settings.

Common setup guidance:

- Treat the customer gateway endpoint as the OpenAI-compatible base URL where supported.
- Use a pilot API key issued by the gateway dashboard.
- Use the model slug configured in **Routes** and granted in **Access**.
- Store keys in approved local or team-managed secret locations.
- Start with a small pilot group before broad rollout.

## Common admin workflows

### Add a new model route

Use this when a new SGL worker or external model endpoint is ready. Create the route, validate health, grant access, test with an admin key, and roll it out to pilot users.

### Rotate a provider credential

Use this when a route credential changes. Update the credential through the approved secret path, verify route health, send a test request, and remove the old credential after validation.

### Revoke a user API key

Use this when a key is compromised, stale, or no longer needed. Revoke the key, issue a replacement if needed, and confirm the affected user or team updates their coding-agent configuration.

### Add a pilot team

Use this when expanding from initial admins to a broader engineering group. Invite users, create keys, grant model access, set limits, share setup instructions, and monitor usage during rollout.

### Review and adjust limits

Use this during pilot expansion or after capacity changes. Review usage, latency, errors, and GPU utilization before increasing limits.

### Disable a route

Use this when a worker is unhealthy or a provider should stop receiving traffic. Mark the route inactive, validate fallback behavior if available, and communicate expected user impact.

## Admin handoff

At the end of setup, Subconscious and the customer should complete an admin handoff.

Handoff checklist:

- Customer admins can sign in.
- Admin users and pilot users are configured.
- Initial API keys are created and stored.
- Initial SGL worker routes are active and healthy.
- Model access is granted.
- Limits are configured.
- Test request succeeds.
- Coding-agent setup is validated for at least one pilot user.
- Usage is visible.
- Monitoring and logging ownership is confirmed.
- Support and escalation process is understood.
