Subconscious uses [Distr](https://distr.sh/docs/) to help distribute and operate customer-hosted deployments. Distr is built for delivering Docker, Helm, and OCI artifacts into self-managed, assisted self-managed, bring-your-own-cloud, air-gapped, and edge environments.

During onboarding, your Subconscious FDE provisions a customer portal for your organization. The portal gives your team a place to view deployment instructions, inspect available artifacts, manage deployments, generate access tokens, and review release resources.

## What Distr is used for

The distribution platform supports the software distribution plane. This is separate from the runtime plane that serves inference traffic.

Subconscious uses the distribution plane to deliver:

- Helm charts and container images.
- Customer-specific deployment instructions.
- Registry access and artifact metadata.
- Deployment status for assisted self-managed deployments.
- Release notes, version metadata, and upgrade guidance.
- Vulnerability reports and release evidence where available.
- Customer-managed deployment secrets when enabled for the deployment.

The runtime plane is the system your engineers use. The distribution plane is the mechanism for delivering, updating, and supporting that system.

## Customer portal

The customer portal is the customer-facing workspace for deployment and distribution workflows.

Customers use it to:

- View deployment instructions.
- Inspect available artifacts.
- Generate registry access tokens.
- Manage deployment secrets when enabled.
- Review release notes and version resources.
- Review vulnerability reports when attached to releases.
- View deployment status in assisted self-managed mode.

## Artifact registry

The artifact registry is where licensed Subconscious artifacts are made available to your organization. It can include Docker images, Helm charts, and other OCI artifacts.

Depending on your policy, your team may:

- Pull artifacts directly from the Distr registry.
- Mirror artifacts into an internal registry.
- Scan artifacts with internal security tooling before promotion.
- Promote approved artifacts through an internal GitOps or release process.

Traditional self-hosted deployments usually rely more heavily on customer-owned promotion workflows. Assisted self-managed deployments can use the portal and agent to simplify installation and updates.

## Deployment agents

In assisted self-managed deployments, a Distr Kubernetes agent can run in the target environment. The agent manages Helm lifecycle operations locally and reports deployment status back to the portal.

The agent is useful when:

- You want a guided installation path.
- You want shared visibility into deployment status.
- You want customer-approved or scheduled update workflows.
- You want your FDE to help manage deployment operations without direct cloud access.

The customer should still define the namespace, RBAC scope, network policy, secrets policy, and approval process for agent installation.

## Version resources

Version resources are customer-visible materials attached to a release. They help the customer review, approve, and audit deployment changes.

Resources may include:

- Release notes.
- Vulnerability scan reports.
- SBOMs where available.
- Image digests.
- Helm chart version.
- Checksums.
- Compatibility notes.
- Rollback instructions.

These materials support internal change management and security review. For compliance framing, see [Compliance](/on-prem/trust-center/compliance).

## Secrets

Secrets are configured based on the deployment method and customer policy.

In assisted self-managed mode, deployment secrets may be managed through the customer portal when enabled for the organization. In traditional self-hosted mode, customers may prefer to store secrets entirely in their own secret manager and reference them from Helm values, Kubernetes Secrets, or an external secrets operator.

Common secrets include:

- Registry access tokens.
- Gateway admin credentials.
- Model provider credentials.
- Database and cache credentials.
- Observability export credentials.
- TLS certificate references.

Runtime route credentials and API keys are covered in [Gateway setup](/on-prem/deployments/gateway-setup).
